Orbit Wars: Timing-Aware Expansion And Commitment Planning
Reference: this notebook was prepared with reference to Orbit Wars 2026 - Starter by sigmaborov on Kaggle.

🗺️ Big Picture
This notebook exports a compact submission.py agent built around one simple idea: expand safely, then coordinate pressure with better timing than the opponent.

🌍 Build the first economy wave on nearby static neutrals.
☀️ Treat sun safety as a hard physical constraint, not a soft heuristic.
🎯 Aim at future planet and comet positions with boundary-aware intercept timing.
🛡️ Reuse one combat model for both defensive reserves and target capture estimates.
🤝 Track commitments by arrival time and allow small two-source finishes when one planet alone is not enough.
🚚 Use rear planets as logistics hubs instead of leaving ships idle in the backline.
The notebook follows the same order as the agent itself: legal movement geometry, moving-target interception, future combat evaluation, commitment-aware dispatch, and a final smoke run.

Timing-Aware Expansion Strategy Map
Safe geometry, moving-target intercepts, shared combat simulation, and ETA-aware coordination drive the policy.
🌍 Core Plan
- Open on safe static neutrals
- Keep reserve instead of emptying planets
- Coordinate arrivals by ETA
- Convert rear planets into logistics
🧱 Physics And ETA
- Launch from the source boundary
- Hit the target circle, not the center
- Reject any sun-crossing straight shot
- Use one timing model everywhere
🎯 Intercept Search
- Forecast orbit and comet motion
- Iterate toward a future intercept
- Accept small turn-rounding tolerance
- Search later safe windows if blocked
🛡️ Combat And Reserves
- Aggregate same-turn arrivals by owner
- Cancel top-two attacks before garrison
- Reuse the same simulator for defense
- Add a small proactive frontier buffer
🤝 Coordination And Launch
- Keep target commitments ETA-aware
- Allow small two-source finishes
- Preserve separate launches and timing
- Score follow-up attacks and rear logistics
🧭 System Logic
Layer	Job	Main Idea
1. Geometry	keep launches physically legal	boundary-aware launch distance, target-circle hit timing, sun-safe routing
2. Prediction	aim at moving objects	rotating planets, comets, and near-fixed-point intercept search
3. Combat Model	evaluate future fights consistently	same-turn top-two cancellation for both defense and capture
4. Coordination	avoid timing mistakes	ETA-aware commitments and small two-source finish plans
5. Dispatch	turn value into moves	opening bias, 2-player/4-player split, rear logistics
6. Verify	confirm export safety	compile, import, and smoke-test the rebuilt artifact
The notebook is intentionally strategy-first: every section explains the idea in plain language before the matching code cell rebuilds that part of submission.py.

🧱 Geometry And Movement Primitives
This section defines the physical rules the agent uses before it ever scores a target.

Core logic:

📏 separate static planets from rotating ones,
🚀 convert fleet size into real travel speed,
🌗 launch from the source boundary and measure travel until the target circle is hit,
☀️ reject straight-line routes that cross the sun,
⏱️ expose one travel-time model that later prediction and evaluation code will reuse.
In short, this section answers is this launch physically legal, and how long does it really take?.

🎯 Trajectory And Prediction Helpers
This section builds the timing layer that sits on top of the movement primitives.

Core logic:

🛰️ predict where rotating planets and comets will be in future turns,
🎯 solve moving-target intercepts against the target circle instead of only the target center,
☀️ if the current shot is sun-blocked, search future windows where a straight launch becomes valid,
⏱️ accept small turn-rounding error instead of requiring a perfect fixed-point equality.
In short, this section tries to answer when can we hit that target with one legal straight-line launch?.

🛡️ Defense And Evaluation Helpers
This section defines the evaluation layer: how many ships are safe to spend, how future battles reshape a target, and which targets are worth contesting.

Core logic:

🧮 resolve same-turn arrivals with the same top-two cancellation rule used by the environment,
🛡️ estimate the minimum ships each owned planet must keep to survive projected pressure,
🪫 compute safe spend from exact defensive reserve rather than raw current garrison,
⏱️ compare reaction times for neutral targets,
🌐 measure indirect value through frontier distance, nearby production, and enemy density,
👥 keep 2-player and 4-player pressure handling separate through explicit constants.
In short, this section answers two questions before attacking: can we afford to send ships? and what will the target actually look like when we arrive?.

Appending to submission.py
🚀 Main Strategy In agent(obs)
This section combines the earlier helper layers into the final decision policy.

Core logic:

🧭 read the board and separate owned planets, enemy planets, and neutral expansion targets,
🛡️ compute reserve requirements and true spendable ships for every owned planet,
🎯 build source-level attack options only after a first intercept solve,
🤝 keep target commitments ETA-aware so only on-time help is counted,
🚀 allow small same-turn two-source attacks when one planet alone cannot finish the job,
🚚 move leftover rear ships toward the most relevant frontier without merging launches and changing timing.
In short, this section decides which targets to attack, when to attack them, and how many ships to commit.
